2. Reverting Pushed Code (Undoing Mistakes Safely)
If you pushed code to GitHub and it broke your Render deployment, you have two primary ways to undo it. In a shared team environment, Method A is the industry gold standard because it doesn't rewrite history.

Method A: git revert (The Safe, Professional Way)
Instead of deleting the bad commit from history, git revert takes the exact opposite of your changes and creates a brand-new commit on top to cancel it out. This ensures your team members don't get sync errors.

Bash
# 1. Revert the bad commit by targeting its ID
git revert 8a14161

# 2. A text editor might pop up asking for a commit message. Save and close it.
# 3. Push the new "undo" commit to the cloud
git push origin main
Your GitHub Actions pipeline will run, Render will see the revert commit, and your server will automatically roll back to the working version safely!

Method B: git reset --hard (The Destructive Way)
Use this only if you are the only one working on the branch, because it completely obliterates commits as if they never existed.

Bash
# Move your local branch history backwards to a specific working commit ID
git reset --hard 3863d8c

# Because your local history is now BEHIND GitHub, a normal push will be rejected.
# You must force-overwrite the cloud repository:
git push origin main --force

Now, tell Git to create a brand-new commit that reverses everything inside that bad commit:

Bash
git revert a1b2c3d
(Replace a1b2c3d with the actual Commit ID from your terminal log).

📝 Note on the Text Editor Pop-up:
When you run this, Git will automatically open a text editor inside your terminal window (usually a tool called Vim or Nano) asking you to confirm the commit message.

If it opens Vim (you'll see columns of ~ marks), type :wq and hit Enter to save and exit.

If it opens Nano (you'll see command options at the bottom), press Ctrl + O then Enter to save, and Ctrl + X to exit.
